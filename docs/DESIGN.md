# pft-validator - Design

This repo is a thin presentation layer. The interesting design is in WHAT
it presents, WHERE the data comes from, and HOW the system stays
self-updating without a real backend.

## Three properties in one repo

The site is one origin (`pft.permanentupperclass.com`) but serves four
unrelated sub-properties plus the validator's root attestation:

- `/` - the validator dashboard (WebSocket-driven, real-time)
- `/scanner/` - the LLM Convergence Scanner ranked table (file-driven,
  refreshed by the `puc-trading` deploy script)
- `/herald/`, `/subs/`, `/lens/` - validator-network feeds, refreshed
  hourly by the `pf-scout-bot` indexer
- `/forensics/`, `/alpha/`, `/research/` - manually-curated content

They share an origin because GitHub Pages bills per-repo, the validator
already needs the apex domain for `.well-known/xrp-ledger.toml`
attestation, and these properties are operationally co-managed.

Each sub-property is intentionally **flat HTML + one JSON fetch**. No
JavaScript build step, no framework, no SSR. The point is that GitHub
Pages can serve them with zero infrastructure.

## Why the data lives elsewhere

Static GitHub Pages can't query a database, run a model, or hit the XRPL
WebSocket. Everything dynamic comes from a JSON file in the same path as
the HTML.

That file is regenerated by a separate process on the `pf-scout-bot` box
and pushed to this repo. The site stays static; the freshness comes from
the commits.

## The hourly pipeline

`/home/ubuntu/pf-scout-bot/deploy/hourly-pipeline.sh` runs as cron at
`0 * * * *`. Each step exports one feed and commits it here:

1. `export-audit.sh` -> audit feed
2. `export-graph.sh` -> agent graph
3. `export-health.sh` -> validator health snapshot
4. `export-auth.sh` -> auth audit
5. `export-subs.sh` -> SUBS registry
6. `export-herald.sh` -> Hive Herald edition
7. `export-herald-latest.sh` -> herald/latest.json

Each step is `set -e` and commits with a deterministic message like
`Health snapshot 2026-05-14T20:05`. The push uses a fine-grained PAT that
lives in `~/pf-scout-bot/.env`.

Daily Herald delivery is separate: `deploy/deliver-herald.sh` runs at
`5 0 * * *` and posts the day's edition on-chain to paying subscribers
(via the SUBS protocol). The published page in `/herald/` is the
non-paywalled mirror.

### Failure modes

If feeds go stale (you see no new commit for >2h), one of:

- `pf-scout-bot` host is down.
- The push token in `~/pf-scout-bot/.env` has been rotated and the
  scripts haven't been updated. Look for "remote: Permission denied" in
  `/var/log/syslog` or in `/tmp/cron-hourly.log`. The deploy scripts use
  `set -e`, so the first failing step halts the rest. Health/auth/subs
  going stale while audit/graph keep updating is the classic signal that
  the token died after the audit/graph steps -- those use a different
  pathway.
- Chain index corruption. Symptom: indexer step fails with sqlite
  malformed-record. Fix: `sqlite3 .recover` into a fresh DB, swap in.

## The scanner sub-property

`/scanner/` is fed by the `puc-trading` runtime, not by the hourly
pipeline. Specifically:

1. `puc-trading/scanner/run_live_scan.py` connects to IB Gateway and
   writes `~/pft-validator/scanner/scan-results.json`.
2. `puc-trading/scripts/deploy-scanner-results.sh` validates that file
   (shape + freshness + secret-pattern scan) and commits + pushes to
   THIS repo.
3. `/scanner/index.html` fetches the JSON on page load.

Recommended cadence is daily (not hourly) -- convergence does not move
that fast. The deploy script has a `DEPLOY_PUSH=1` env gate so a cron can
be enabled without changing the script.

## The validator dashboard at /

`index.html` is the only page that does live WebSocket. It tries each URL
in `CONFIG.wsEndpoints`:

```
wss://pft.permanentupperclass.com:51233
wss://pft.permanentupperclass.com:6006
```

These point at the validator node's XRPL WebSocket. If the node is down,
the page degrades gracefully (shows last known state).

## Things that look static but are managed by other systems

- `/herald/latest.json` and dated edition files are written by
  `export-herald-latest.sh` per hour and by `deliver-herald.sh` once a day.
- `/subs/` registry feeds are managed by `export-subs.sh`.
- `/lens/` is managed by the hourly Lens update step.
- `/.well-known/xrp-ledger.toml` is currently manual. Update on validator
  key rotation (use the `validator-keys` tool).

## Things that are real submissions

`/alpha/<task-slug>/` contains the published deliverables for every PFT
task that has shipped publicly. Each subdirectory has an `index.md`
(rendered by GitHub Pages) plus any supporting artifacts (gists, code).
These are added by hand at submission time.

## Repo hygiene

- IB Gateway sometimes scatters junk into the working tree (`jts.ini`,
  `launcher.log`, `dgpdjeilgkccmlebkicghonjccocflnajmhbcnmh/`). The
  `deploy-scanner-results.sh` script in `puc-trading` runs an
  `ensure_gitignore` step that idempotently appends these patterns.
- Hourly pipeline scripts use whitelisted file paths in `git add` --
  never `git add -A`.
- Push token is provisioned out-of-band. Do NOT commit
  `~/pf-scout-bot/.env` or any other secret.
